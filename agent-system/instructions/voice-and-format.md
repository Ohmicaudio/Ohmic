# Voice and Format

This file defines how shared agent docs should sound and how they should be structured.

## Voice Rules

Use a voice that is:

- calm
- plainspoken
- specific
- operational

Write like a teammate leaving trustworthy notes for another teammate.

## Sentence Rules

- prefer short declarative sentences
- prefer concrete nouns over vague abstractions
- say what is true now, not what might vaguely be true
- avoid hype, filler, and sales language

## Preferred Verbs

Use direct verbs such as:

- `is`
- `uses`
- `owns`
- `tracks`
- `needs`
- `blocks`
- `depends on`
- `points to`
- `promote`
- `archive`

Avoid mushy verbs such as:

- `leverages`
- `utilizes`
- `facilitates`
- `optimizes` when you only mean `improves`

## Point Of View

- write memory and project docs in neutral operational voice
- write handoffs in factual past/present voice
- avoid diary-style writing
- avoid speaking as if the document itself is the actor

Good:

- `The semantic index is running from B:\\ohmic\\tools\\semantic-index.`
- `Reference material should live under B:\\ohmic and below.`

Bad:

- `We are super excited to leverage the semantic layer for enhanced continuity.`

## Allowed Status Terms

Use the canonical labels from `memory/terminology.md`:

- `canonical`
- `working`
- `reference`
- `deprecated`
- `archive`

Do not invent near-synonyms unless there is a real new state.

## Shared Section Names

Prefer these exact section names when relevant:

- `Identity`
- `Working Location`
- `Current Truth`
- `Known Drift`
- `Active Priorities`
- `Always-Load Context`
- `Skills`
- `Keywords`
- `Access`
- `Watchouts`
- `Learned Lessons`
- `Next Move`
- `First Read`

## Shared Field Names

When capturing compact data, use these labels:

- `skills`
  - active or durable competencies relevant to the work
- `keywords`
  - retrieval or routing terms
- `access`
  - credentials, tools, or verified capabilities available in the workspace
- `watchouts`
  - fresh problems, fragile areas, recent mistakes
- `lessons`
  - patterns learned from repeated work or repeated failures
- `next move`
  - the clearest next action

## Formatting Rules

- use flat bullets, not nested outlines, unless the document already depends on indentation
- keep bullet items short and standalone
- use one bullet per fact where possible
- keep headings stable across files so retrieval and scanning are predictable

## Scope Rules

- `short-term` should feel like a live control panel
- `mid-term` should feel like an initiative and lessons register
- `long-term` should feel like durable policy and truth
- project overlays should feel like a current operating brief
- handoffs should feel like a concise baton pass
