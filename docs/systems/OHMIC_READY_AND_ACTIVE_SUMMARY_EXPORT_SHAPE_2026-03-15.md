# Ohmic Ready And Active Summary Export Shape

Date: 2026-03-15
Status: working contract

## Purpose

Define the minimal generated JSON export shape for:

- `ready_tasks.json`
- `active_claims.json`

These files are convenience summaries for dashboards and wrappers.

They are not canonical authority.

## Source Of Truth Rule

Canonical live truth stays in repo-backed files:

- `agent-system/requests/ready/`
- `agent-system/jobs/active/`

The JSON exports are derived views.

If the JSON summaries disagree with the repo-backed directories:

1. trust the repo-backed directories
2. mark the summary stale
3. regenerate the summary

Do not let consumers silently promote the summaries into authority.

## Producer

These exports should be generated by the same reconciliation/wrapper family that
refreshes `agent_state.json`.

Related contracts:

- `docs/systems/OHMIC_STATE_RECONCILIATION_PASS_2026-03-15.md`
- `docs/systems/OHMIC_AGENT_RUNTIME_JSON_CONTRACT_2026-03-15.md`
- `docs/systems/OHMIC_JSON_AGENT_LOOP_MODEL_2026-03-15.md`

## Shared Envelope

Both files should use the same summary envelope pattern:

```json
{
  "generated_at": "2026-03-15T20:05:00Z",
  "source": {
    "kind": "repo_summary",
    "path": "B:\\ohmic\\agent-system\\requests\\ready"
  },
  "staleness": {
    "status": "fresh",
    "reason": null
  },
  "count": 2,
  "items": []
}
```

## Common Fields

Required top-level fields:

- `generated_at`
- `source`
- `staleness`
- `count`
- `items`

### `source`

Purpose:

- identify what stronger source produced the summary

Shape:

```json
{
  "kind": "repo_summary",
  "path": "B:\\ohmic\\agent-system\\requests\\ready"
}
```

### `staleness`

Purpose:

- let consumers show honest freshness without pretending the summary is always
  current

Shape:

```json
{
  "status": "fresh",
  "reason": null
}
```

Allowed `status` values:

- `fresh`
- `stale`
- `unknown`

Recommended `reason` values:

- `null`
- `reconciliation_pending`
- `source_mismatch`
- `summary_missing`
- `generation_failed`

## `ready_tasks.json`

Purpose:

- expose a lightweight list of currently ready tasks

### Item Shape

Each item should contain summary-only task metadata:

- `task_id`
- `title`
- `priority`
- `project`
- `status`

Recommended fields:

- `date`
- `owner`
- `claim_id`

Example:

```json
{
  "generated_at": "2026-03-15T20:05:00Z",
  "source": {
    "kind": "repo_summary",
    "path": "B:\\ohmic\\agent-system\\requests\\ready"
  },
  "staleness": {
    "status": "fresh",
    "reason": null
  },
  "count": 2,
  "items": [
    {
      "task_id": "2026-03-15-define-ready-and-active-summary-export-shape",
      "title": "Define Ready And Active Summary Export Shape",
      "priority": "low",
      "project": "ohmic",
      "status": "ready",
      "date": "2026-03-15"
    },
    {
      "task_id": "2026-03-15-define-dashboard-status-card-mapping",
      "title": "Define Dashboard Status Card Mapping",
      "priority": "low",
      "project": "ohmic",
      "status": "ready",
      "date": "2026-03-15"
    }
  ]
}
```

### What To Leave Out

Do not copy full request bodies into the summary export:

- long goals
- long focus sections
- acceptance prose
- request notes/history

If a consumer needs the full task body, it should open the Markdown file.

## `active_claims.json`

Purpose:

- expose a lightweight list of live claims for dashboards and wrappers

### Item Shape

Required fields:

- `claim_id`
- `task`
- `owner`
- `project`
- `status`
- `started`
- `expires`

Recommended fields:

- `files`
- `request_path`

Example:

```json
{
  "generated_at": "2026-03-15T20:05:00Z",
  "source": {
    "kind": "repo_summary",
    "path": "B:\\ohmic\\agent-system\\jobs\\active"
  },
  "staleness": {
    "status": "fresh",
    "reason": null
  },
  "count": 1,
  "items": [
    {
      "claim_id": "20260315T135409Z-7cc6bb0d",
      "task": "define ready and active summary export shape",
      "owner": "d",
      "project": "ohmic",
      "status": "active",
      "started": "2026-03-15T13:54:09Z",
      "expires": "2026-03-15T15:54:09Z",
      "files": [
        "B:\\ohmic\\docs\\systems\\OHMIC_READY_AND_ACTIVE_SUMMARY_EXPORT_SHAPE_2026-03-15.md",
        "B:\\ohmic\\agent-system\\requests\\ready\\2026-03-15-define-ready-and-active-summary-export-shape.md"
      ]
    }
  ]
}
```

### What To Leave Out

Do not mirror the full claim file body if it is not needed for UI status:

- long notes
- duplicate task prose
- repo content unrelated to the claimed scope

## Consumer Rules

Consumers should treat these files as summary surfaces only.

Minimum consumer behavior:

1. show `count`
2. show a freshness badge from `staleness.status`
3. if `staleness.status != fresh`, avoid strong claims like "queue is current"
4. allow drilling back to repo-backed files when more detail is needed

## Failure And Missing-File Rules

If the summary file is missing:

- render `staleness.status = unknown`
- do not infer `count = 0`

If generation fails:

- preserve the last known file if available
- set `staleness.status = stale`
- set a non-null `reason`

If source counts disagree:

- prefer the repo-backed directories
- treat the JSON summary as stale until regenerated

## Guardrails

- do not write user decisions into these files
- do not use them as the only queue/claim read path for reconciliation
- do not let dashboards mutate queue truth through these summary exports
- do not stuff full Markdown request bodies into the items arrays
