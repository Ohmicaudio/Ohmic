# Ohmic Next 2 Days Working Plan

## Objective

Use the next 48 hours to turn the current software and content base into a
clearer revenue engine while locking the planning surfaces needed for hardware
to start cleanly.

## Why Now

- the migration chaos is finally low enough that direction work will stick
- static content, app/runtime, and hardware planning now have separate homes
- software needs to start paying for hardware instead of waiting on hardware to
  justify software
- the team now needs one execution packet, not more scattered good ideas

## Scope In

- finalize planning language and roadmap baseline
- keep static graphics and quality work moving in small reusable families
- verify runtime and static boundaries after the current graphics wave
- define the first software revenue surfaces and funnel
- start structured hardware research intake in the new hardware repo

## Scope Out

- large new firmware implementation work
- first-pass schematic capture
- physical hardware layout
- broad brand redesign
- paid ads or large outbound campaigns

## Owners

- `primary execution agent`: roadmap packet, offer structure, marketing scaffold,
  boundary verification, coordination cleanup
- `parallel graphics/content agent`: graphics microtasks, post-graphics page QA,
  placeholder cleanup
- `user`: approve direction, supply research/chat dumps, decide packaging and
  positioning when tradeoffs are real

## Day 0 To Day 1

### 1. Lock the planning language

Why:

- prevents `proposal` and `approval` from drifting back into vague chat terms

How:

- publish definitions
- publish 2-day working plan
- publish longer roadmap proposal

Output:

- roadmap packet in `B:\ohmic\docs\roadmap`

Exit criteria:

- the planning docs exist
- the terms are explicit enough to use operationally

### 2. Keep the active graphics/content wave moving

Why:

- static quality work is already in motion and directly improves acquisition

How:

- continue family-based graphics tasks
- keep claims visible
- close queue items immediately after each asset family lands

Output:

- more placeholder graphics replaced in `ohmic-audio-static-content`
- queue state stays truthful

Exit criteria:

- at least one more reusable family lands cleanly
- no reopened parity churn

### 3. Define the first software revenue ladder

Why:

- software must fund hardware
- the platform already has enough tools/content/runtime surface to package

How:

- define three layers:
  - `free`: SEO tools, static guides, lead capture
  - `pro`: advanced design/measurement workflows, saved projects, exports,
    support priority, premium tools
  - `hardware-linked`: control plane, device workflows, premium measurement and
    commissioning tied to AmpLab hardware

Output:

- one concrete offer ladder document or section inside the roadmap proposal

Exit criteria:

- there is a named first monetization structure
- first paid surface candidates are listed

## Day 1 To Day 2

### 4. Verify the app/static boundary after the current graphics wave

Why:

- route or asset confusion will poison software packaging and traffic capture

How:

- run the queued static route smoke check
- run the app/static boundary verification task
- confirm the runtime app is acting like runtime and the static repo is acting
  like static

Output:

- verified route list
- explicit notes on any remaining boundary leaks

Exit criteria:

- the boundary check is complete
- remaining leaks are either fixed or queued

### 5. Turn hardware into structured intake instead of loose ambition

Why:

- big hardware design goes badly when the first artifact is the schematic

How:

- begin filling research-intake files in:
  - `amplab-power-master`
  - `dsp-wireless-controller`
  - `rta-spl-meter`
- start turning the first collected notes into:
  - `requirements.md`
  - `block-diagram.md`
  - `power-tree.md`
  - `interface-matrix.md`

Output:

- seeded hardware planning packets with real source material

Exit criteria:

- intake is no longer just in chat or memory
- at least one primary module has enough material to begin requirements sorting

### 6. Stage the first marketing and social operating scaffold

Why:

- the product needs acquisition loops, not just product surfaces

How:

- define initial channels and jobs:
  - static SEO pages for search capture
  - short-form clips/screenshots for social proof
  - build logs and hardware planning posts for narrative continuity
  - community/support surfaces for trust and feedback

Output:

- a lightweight channel and content cadence section in the roadmap proposal

Exit criteria:

- the first channel mix is written down
- content jobs are tied to real product surfaces

## Deliverables Due Inside 48 Hours

- planning language defined and durable
- 2-day working plan written
- year-one roadmap proposal written
- first monetization ladder defined
- hardware intake packets started with real source material
- active graphics wave still progressing without coordination churn

## Approval Questions

- does the `free / pro / hardware-linked` ladder fit the intended business shape
- should the first paid push focus on design workflows, measurement workflows,
  or premium support first
- which hardware module should get the first deep requirements pass

## Exit State

If this 2-day plan succeeds, Ohmic should end the window with:

- one stable planning language
- one visible year-one direction
- one software-first revenue thesis
- one clean hardware planning intake path
- one still-moving static/content quality engine
